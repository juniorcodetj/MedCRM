1. Резюме аудита

Backend и архитектурные документы описывают серьезную SaaS CRM/МИС платформу для частных клиник: multi-tenant, RBAC, Patient CRM, Smart Scheduling, Receptionist Workplace, а также заготовки под финансы, EMR, коммуникации, BI, склад и интеграции. Фронтенд пока выполняет роль MVP-оболочки и не соответствует масштабу продукта.

Область Найдено сейчас Что требуется в дизайне
Визуальная система Глобальный CSS, Arial, простые панели, нет токенов компонентов. Цельная медицинская SaaS-айдентика, дизайн-токены, компоненты, статусы, таблицы, формы.
Навигация Левое меню из четырех пунктов: Dashboard, Пациенты, Расписание, Регистратура. Модульная навигация с группировкой, активными состояниями, поиском, user/tenant контекстом.
Dashboard Отображает JSON с количеством модулей и permissions. Операционная панель клиники: день, поток пациентов, загрузка врачей, деньги, задачи, риски.
Пациенты Список ссылок, простой поиск, короткая форма создания, базовая карточка пациента. CRM-реестр с фильтрами, дублями, сегментами, timeline, контактами, статусами и быстрыми действиями.
Расписание Линейный список записей на сегодня и форма создания записи. Календарь день/неделя, ресурсы, врачи, слоты, конфликты, быстрый booking flow.
Регистратура Горизонтальная kanban-доска статусов и очередь. АРМ администратора: today board, быстрый check-in, очередь, звонки, мини-карточка пациента, realtime.
Состояния интерфейса Минимальные loading/error сообщения, нет empty states, skeletons, toasts, confirmations. Полный набор состояний для загрузки, ошибок, пустых списков, прав доступа, offline/retry и realtime.
Вывод: дизайн нужно проектировать не как набор красивых страниц, а как операционную рабочую систему клиники: плотную, спокойную, быструю, с высоким доверием к данным и минимальным числом переключений.

2. Продуктовый контекст

MedCRM предназначен для частных клиник и должен совмещать CRM, расписание, регистратуру, коммуникации, аналитику и будущие медицинские модули. Основной пользователь работает под давлением времени: звонки, пациенты у стойки, переносы записей, оплата, очередь, вопросы врачей.

Администратор

Главный пользователь MVP. Ему нужны быстрый поиск, запись пациента, check-in, очередь, статусы визитов и минимум переходов.

Управляющий клиники

Смотрит поток пациентов, загрузку филиалов и врачей, финансовые показатели, проблемные зоны и качество обработки.

Врач и медицинский персонал

Получает контекст визита, статус пациента, кабинет, услугу, историю и будущий переход в EMR/рабочее место врача.

Ключевые продуктовые принципы

  • Интерфейс должен быть операционным, а не маркетинговым: без hero-блоков внутри приложения и без декоративной перегрузки.
  • Главные действия должны быть доступны за 1-2 клика: найти пациента, создать запись, подтвердить, отметить приход, отменить, перенести.
  • Система обязана ясно показывать статус данных: филиал, дата, realtime-соединение, активный пользователь, права, модуль.
  • Медицинская тематика требует сдержанного, доверительного визуального языка: чистые поверхности, высокая читаемость, аккуратные статусы.
  • Дизайн должен масштабироваться под новые модули: finance, communications, EMR, BI, inventory, integrations.

3. Текущее состояние фронтенда

Технологии

Фронтенд использует Next.js 16 App Router, React 19, TypeScript, TanStack Query, Zod, Socket.IO Client. В архитектурных документах запланированы TailwindCSS, shadcn/ui, Zustand, next-intl, PWA, ECharts, но в текущем frontend/package.json TailwindCSS, shadcn/ui, icons, charting и i18n еще не подключены.

Фактические экраны

Маршрут Текущая роль Основной дефицит UX/UI
/auth/login Вход по tenant code, email, password. Нет брендовой доверительной страницы, подсказок безопасности, MFA-состояния, локализации.
/dashboard Проверка bootstrap payload. Нет реального dashboard: KPI, задач, визитов, динамики, предупреждений.
/patients Поиск, список пациентов, создание пациента. Нет табличного реестра, фильтров, сортировки, дубликатов как полноценного workflow.
/patients/[id] Карточка пациента и смена статуса. Нет CRM-профиля, истории, контактов, визитов, документов, заметок и быстрых действий.
/schedule Список записей на сегодня и создание записи. Нет календарной сетки, ресурсов, фильтров по врачу/кабинету, конфликтов, drag/reschedule.
/reception Kanban по статусам и очередь. Статусы технические, доска перегружена 7 колонками, нет приоритетов, звонков, оплаты, sticky context.

Главные UX-риски

  • Пользователь видит технические статусы на английском, а не понятные операционные состояния.
  • Формы не объясняют ошибки до отправки и не дают быстрых альтернатив при конфликте записи.
  • Списки представлены как карточки/ссылки, что плохо масштабируется для клиники с большим потоком пациентов.
  • Дизайн не показывает realtime-доверие: непонятно, свежие ли данные, идет ли синхронизация, кто изменил статус.
  • Нет единого языка статусов между расписанием, регистратурой, карточкой пациента и будущими модулями.

4. Дизайн-направление

Визуальная теза: спокойный медицинский операционный интерфейс с плотной информационной сеткой, чистыми белыми поверхностями, графитовым текстом, бирюзовым основным акцентом и ограниченной системой статусных цветов.

Характер интерфейса

  • Современный B2B SaaS, ближе к Linear/Notion/Intercom по чистоте, но плотнее по данным.
  • Меньше декоративных карточек, больше таблиц, панелей, split-view и контекстных drawers.
  • Ощущение медицинской надежности: точность, порядок, видимость статуса, отсутствие визуального шума.

Интерактивная теза

  • Мягкие transitions для drawer/modal, смены статуса и появления realtime-событий.
  • Drag-and-drop только там, где он ускоряет работу: расписание и today board.
  • Hover/focus состояния помогают сканировать строки и карточки, но не меняют layout.

5. Информационная архитектура

Навигация должна отражать модульность платформы, но не превращаться в длинный список. На MVP-этапе рекомендуется сгруппировать разделы по рабочим зонам.

Группа Пункты Назначение
Операции Операционная панель, Регистратура, Расписание Ежедневная работа клиники и поток пациентов.
CRM Пациенты, Сегменты, Коммуникации База пациентов, удержание, история контактов, рассылки.
Клиника Филиалы, Сотрудники, Кабинеты, Услуги Организационная структура и ресурсы записи.
Управление Финансы, BI, Склад, Интеграции Административные и управленческие модули.
Система Пользователи, Роли, Настройки, Audit log Tenant, RBAC, безопасность и конфигурация.

Shell приложения

  • Левый sidebar: логотип, группы навигации, активный пункт, компактный режим, права доступа.
  • Верхняя панель: поиск/command palette, филиал, дата/смена, realtime status, уведомления, профиль пользователя.
  • Рабочая область: max-width не ограничивает операционные таблицы; важные экраны используют всю ширину.
  • Правый drawer: быстрый просмотр пациента, записи, счета, звонка или события без ухода со страницы.

6. ТЗ по экранам

6.1 Login /auth/login

Цель: быстро и безопасно войти в клинику/tenant.

  • Слева или сверху: MedCRM, короткая формулировка продукта для клиники, доверительный медицинский визуальный фон без стоковой перегрузки.
  • Форма: код клиники, email, пароль, submit, состояние загрузки, ошибка, восстановление доступа.
  • Поддержать future-state для MFA: экран ввода кода, повторная отправка, истекший код.
  • После ошибки не сбрасывать заполненные поля, показывать понятную причину без раскрытия лишней security-информации.
6.2 Операционная панель /dashboard

Цель: дать управляющему и администратору обзор текущего дня.

  • Верхний блок: дата, филиал, смена, realtime status, быстрые действия "Новая запись", "Новый пациент".
  • KPI: записи сегодня, пришли, ожидают, отмены/no-show, загрузка врачей, выручка/ожидаемая оплата.
  • Операционный список: ближайшие визиты, пациенты в очереди, проблемные записи, конфликты расписания.
  • Небольшой график загрузки по часам и по врачам; для реализации использовать будущий ECharts или аналог.
  • Виджеты должны иметь empty/loading/error состояния и не показывать технический JSON.
6.3 Пациенты /patients, /patients/[id]

Цель: превратить текущий список в полноценную CRM-базу пациентов.

  • Реестр: таблица с ФИО, кодом, телефоном, статусом, тегами, последним визитом, филиалом, менеджером, LTV/future metric.
  • Поиск: единая строка с поддержкой ФИО, телефона, patient code; фильтры по статусу, филиалу, тегу, дубликатам.
  • Создание пациента: drawer или modal, а не отдельная узкая карточка; валидация телефона и подсказка возможных дублей до сохранения.
  • Карточка пациента: header с ФИО, кодом, статусом, контактами, быстрыми действиями "Записать", "Позвонить", "Сообщение", "Заметка".
  • Внутри карточки: timeline, визиты, контакты, документы/согласия, сегменты, история коммуникаций, будущая финансовая вкладка.
6.4 Расписание /schedule

Цель: заменить линейный список календарным рабочим инструментом.

  • Виды: день, неделя, список; MVP по умолчанию - день по врачам/ресурсам.
  • Фильтры: филиал, врач, отделение, кабинет, услуга, статус.
  • Сетка времени: фиксированная левая шкала, колонки врачей/кабинетов, цветные appointment blocks.
  • Booking drawer: пациент, врач, услуга, дата/время, длительность, заметка; сразу показывать conflicts и alternative slots.
  • Действия по записи: confirm, check-in, cancel, reschedule, открыть пациента; кнопки на русском языке.
6.5 Регистратура /reception

Цель: сделать АРМ администратора главным операционным экраном клиники.

  • Основной экран: today board с колонками "Ожидают", "Пришли", "На приеме", "К оплате", "Завершено", "Отмены/не пришли".
  • Карточка визита: время, пациент, врач, кабинет, услуга, статус, VIP/долг/первичный визит, быстрый check-in.
  • Правая панель: очередь, быстрый поиск пациента, входящие звонки/future placeholder, последние события.
  • Drag-and-drop должен сопровождаться явным visual feedback и confirmation/reason для критичных переходов.
  • Realtime-события показывать как мягкие обновления: строка подсвечивается, появляется "обновлено только что", без резких layout shifts.
6.6 Будущие модули finance, communications, EMR, BI, inventory, integrations

Цель: заложить паттерны, чтобы новые модули выглядели частью одной системы.

  • Каждый модуль получает common page header, filters bar, table/list pattern, empty state, permissions state.
  • Finance: счета, оплаты, задолженности, статусы платежей, связь с визитом и пациентом.
  • Communications: inbox, шаблоны, история SMS/Email/мессенджеров, consent-aware действия.
  • BI: спокойные графики, фильтры периода/филиала, drill-down в таблицы.
  • EMR: визуально отделять клинические данные от CRM-данных и учитывать повышенную чувствительность информации.

7. Дизайн-система

Базовые токены

Категория Требование
Типографика Inter или Geist Sans. Базовый размер 14-15px для рабочих поверхностей, 13px для вторичных метаданных, 20-28px для заголовков страниц.
Сетка 8px spacing scale. Shell: sidebar 248-280px, topbar 56-64px, content padding 20-28px. Таблицы и календарь используют всю ширину.
Радиусы 6-8px для controls и panels. Не использовать чрезмерно круглые карточки для операционного UI.
Тени Минимально: dropdown, modal, drawer. Основная иерархия через фон, границы и spacing.
Иконки Lucide React или аналог: Calendar, Users, Search, Bell, Settings, Stethoscope, CreditCard, Activity, Plus, MoreHorizontal.

Компоненты

Дизайн должен быть готов к реализации через shared/ui и предпочтительно через TailwindCSS/shadcn/ui, как указано в архитектурном ТЗ. Если зависимости еще не добавлены, дизайн все равно должен поставляться как компонентная система, а не как набор уникальных страниц.

Микрокопирайтинг

  • Пользовательские тексты должны быть на русском языке; технические enum значения не показывать без перевода.
  • Кнопки: "Записать", "Подтвердить", "Отметить приход", "Отменить", "Перенести", "Создать пациента".
  • Ошибки должны объяснять действие: "Врач занят в это время. Выберите другой слот" вместо "Conflict".
  • Пустые состояния должны предлагать следующее действие, а не просто писать "Нет данных".

8. Состояния и UX-правила

Состояние Требование
Loading Skeleton для таблиц, календаря, карточек пациента. Кнопки показывают spinner/disabled state без изменения ширины.
Empty Контекстное пустое состояние: "На сегодня записей нет" + действие "Создать запись".
Error Понятное сообщение, retry, техническая деталь скрыта под раскрытием для support/admin.
Permission denied Скрывать недоступные действия; если пользователь попал на route без прав, показать нейтральный экран с объяснением.
Realtime Показывать online/offline индикатор, время последнего обновления, мягкую подсветку обновленных элементов.
Optimistic updates Для смены статусов показывать pending state и rollback при ошибке с toast-уведомлением.
Критичные действия Отмена визита, блокировка пациента, удаление/архивирование требуют confirmation и reason, если это предусмотрено backend.

9. Адаптивность и доступность

Breakpoints

  • Desktop 1440+: полный sidebar, широкие таблицы, календарь с несколькими колонками.
  • Laptop 1024-1439: sidebar может быть компактнее, правые панели открываются drawer-ом.
  • Tablet 768-1023: sidebar collapsible, таблицы переходят в плотный список, календарь ограничивает число колонок.
  • Mobile до 767: основные сценарии read/action only: поиск пациента, check-in, быстрый просмотр очереди, звонок/сообщение.

Accessibility

  • Контраст текста и интерактивных элементов не ниже WCAG AA.
  • Все кнопки с иконками имеют aria-label и tooltip.
  • Drag-and-drop операции имеют keyboard alternative: меню смены статуса.
  • Фокус видим и не обрезается в таблицах, drawer, dialog, sidebar.
  • Цвет не является единственным способом отличить статус: нужен текст/иконка/форма бейджа.

10. Результаты и критерии приемки

Ожидаемые дизайн-результаты

  1. Figma-файл или эквивалентный дизайн-макет с desktop и mobile состояниями.
  2. Компонентная библиотека: tokens, buttons, inputs, tables, badges, drawers, modals, toast, calendar, visit card, patient header.
  3. Макеты ключевых экранов: login, shell, dashboard, patients list, patient profile, schedule, reception board.
  4. Прототипы ключевых сценариев: создать пациента, создать запись, найти пациента, check-in, отменить визит, открыть карточку из очереди.
  5. Документация по статусам, цветам, empty/loading/error/permission состояниям.

Критерии приемки дизайна

  • Интерфейс выглядит как единая медицинская SaaS-система, а не как набор MVP-страниц.
  • Администратор может понять состояние клиники за 10 секунд на dashboard или reception screen.
  • Основные действия администратора доступны без перехода между множеством страниц.
  • Все технические enum/status значения переведены в пользовательский язык.
  • Есть явные состояния для loading, empty, error, permission denied, offline/realtime reconnect.
  • Дизайн покрывает desktop и mobile, при этом mobile не ломает таблицы и рабочие панели.
  • Компоненты готовы к реализации в существующей структуре frontend/modules и frontend/shared.
  • Цветовая система не перегружена одним оттенком и поддерживает медицинские статусы без визуального конфликта.
  • Интерфейс учитывает RBAC, tenant/branch context и модульное подключение разделов.
Важное ограничение реализации: текущий фронтенд еще не содержит TailwindCSS, shadcn/ui, lucide-react, charting и i18n. Если дизайн будет строиться вокруг этих инструментов, отдельной задачей нужно добавить и настроить зависимости, структуру shared/ui, tokens и базовые layout components.

Рекомендуемый порядок реализации после утверждения дизайна

  1. Подключить дизайн-инфраструктуру: tokens, UI primitives, icons, toast, dialog, drawer, data table.
  2. Пересобрать shell: sidebar, topbar, layout, responsive behavior, permissions-aware navigation.
  3. Сделать dashboard и patients как основу visual language.
  4. Переработать schedule и reception, потому что они несут основной операционный UX.
  5. Добавить future-ready паттерны для finance, communications, BI и EMR без реализации лишней бизнес-логики.
