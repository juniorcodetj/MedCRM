1. Сводка

Проект является SaaS CRM/MIS платформой для частных клиник. Реализован монорепозиторий npm workspaces с backend на NestJS, frontend на Next.js App Router, Prisma/PostgreSQL, Redis, Socket.IO и Docker Compose. Архитектурный стиль: модульный монолит с подготовкой к выносу доменов в сервисы.

Основной пользовательский MVP уже покрывает поток: вход в систему, просмотр bootstrap контекста, работа с пациентами, создание записи, отображение записи в расписании и на dashboard регистратуры, realtime-инвалидация через Socket.IO.
Область Фактический статус Комментарий
Auth/RBAC Реализовано JWT access/refresh, refresh cookie, сессии, RBAC guard, branch-aware permissions, TOTP MFA API.
Patient CRM Реализовано расширенно CRUD, поиск, дубли, контакты, теги, семья, документы, notes, timeline, leads, metrics.
Smart Scheduling Реализовано расширенно Записи, статусы, availability, конфликты, ресурсы, лист ожидания, public slots, online booking.
Receptionist Workplace Реализовано backend + UI MVP Today board, queue, check-in, status transitions, calls, invoices, fast booking, realtime.
Organization Structure Backend реализован Филиалы, отделения, сотрудники, должности, кабинеты, оборудование, графики и исключения.
EMR/EHR Backend реализован, UI нет Медкарты, episodes, encounters, подпись, amendment, диагнозы, рецепты, шаблоны, FHIR export.
Finance/Billing Backend реализован, UI нет Смены кассира, оплаты, refunds, wallets, payroll, subscription limits, gateway webhooks.
Communications Backend реализован, UI нет Inbox, messages, templates, rules, campaigns, preferences, SMS/Telegram/chatbot webhooks.
Integration Gateway Backend реализован, UI нет Webhooks, lab orders/results, file metadata/download URL, telephony, devices, FHIR reports.
Business Intelligence Backend реализован, UI нет DWH sync, financial/marketing/operations/doctor dashboards, scheduled reports, realtime cache.
Inventory/Warehouse Backend есть, module registry не включен Код и права есть, но seed не добавляет system module `inventory-warehouse` в `tenant_modules`.
Тесты Не обнаружены В репозитории нет `*.spec.*` или `*.test.*` файлов.

3. Архитектура runtime

Backend applications

Приложение Порт Назначение Фактическое покрытие
api-gateway 3000 Публичный proxy entrypoint. Proxy для `/auth`, `/patients`, `/appointments`, `/availability`, `/slots`, `/services`, `/doctors`, `/reception`, `/socket.io`.
auth-service 3001 Фактический backend-доменный runtime. Содержит auth и все бизнес-модули: CRM, scheduling, reception, org, EMR, finance, communications, integrations, analytics, inventory.
frontend 3002 Next.js App Router UI. Login, protected shell, dashboard, patients, patient details, schedule, reception board.

Core modules backend

Core package Реализовано
core/database PrismaService с lifecycle hooks для connect/disconnect.
core/cache RedisModule с ioredis client provider.
core/audit AuditLoggerService пишет события в `audit_logs` и сохраняет requestId из tenant context.
core/tenancy AsyncLocalStorage context, middleware для requestId/tenant/branch headers, service доступа к context.
core/security Decorators `RequirePermissions`, `RequireModule`, `CurrentUser`, JWT payload types.
core/realtime Socket.IO Redis adapter для масштабируемого realtime.
core/events In-process EventEmitter based event bus с publish/subscribe.
В `auth-service` все доменные модули подключены в одном NestJS runtime. Это соответствует modular monolith, но gateway пока проксирует только MVP-набор путей. Новые backend endpoints доступны напрямую через `auth-service`, но не все доступны через публичный `api-gateway`.

4. Backend-модули

Все бизнес-контроллеры, кроме health и login/refresh, используют JWT/RBAC/module guards. Модульность реализована через Nest modules и Prisma tenant-owned таблицы.

4.1 Auth / RBAC

Компонент Что реализовано
Login Поиск tenant по `tenantCode`, user по tenant+email, argon2 verify, audit success/failed.
Tokens JWT access token, JWT refresh token, refresh token hash в БД, fingerprint rotation.
Sessions `user_sessions`, Redis metadata/fingerprint/revoked keys, logout revokes session.
Bootstrap Tenant, enabledModules, permissions, branches, featureFlags для frontend shell.
Branch-aware RBAC Permissions строятся по active `user_branch_roles`, role permissions и enabled tenant modules.
MFA TOTP secret generation, enable/confirm/disable, MFA pending token, verify flow.
Guards `JwtAuthGuard`, `RbacGuard`, `ModuleEnabledGuard`; read-only mode при `TenantSubscription.SUSPENDED`.

Multi-tenancy

  • Большинство доменных таблиц имеют `tenantId`.
  • Branch-owned данные имеют `branchId` там, где нужно.
  • Уникальные индексы часто scoped по tenant, например `tenantId_code`, `tenantId_email`, `tenantId_patientCode`.
  • RBAC строится через `UserBranchRole`.
PostgreSQL RLS в текущих миграциях не обнаружен. Tenant isolation выполняется на уровне сервисного кода, DTO/guards и Prisma query filters.

8. Безопасность и доступ

Реализовано

  • Argon2 password hashing.
  • JWT access token с `tenant_id`, `branch_ids`, `role_ids`, `permissions`, `session_id`.
  • Refresh token в httpOnly cookie `/auth/refresh`.
  • Refresh token rotation и fingerprint.
  • Redis session cache и revoked marker.
  • RBAC guard по endpoint permissions.
  • ModuleEnabledGuard по `tenant_modules`.
  • Read-only emergency mode при suspended subscription.
  • TOTP MFA API.
  • Helmet, CORS, cookie parser.
  • Audit logging основных действий.

Ограничения

  • Нет rate limiting/bruteforce protection.
  • Нет password policy/history/lockout.
  • Нет OAuth/SSO.
  • Нет RLS на уровне PostgreSQL.
  • TenantContextMiddleware подключен как provider, но `AppModule.configure` пустой.
  • MFA secret хранится как `secretHash`, но фактически это base32 secret, а не hash.
  • Access token frontend хранит в JS cookie, а не httpOnly cookie.
  • Gateway Swagger не отражает proxied бизнес API.

11. Ограничения, несоответствия и риски

Приоритет Наблюдение Почему важно
High Gateway проксирует только MVP endpoints. EMR, finance, communications, integrations, analytics, inventory, organization endpoints не доступны через public gateway без прямого обращения к auth-service.
High `inventory-warehouse` не добавлен в seed system modules. ModuleEnabledGuard заблокирует inventory API для demo tenant, несмотря на наличие контроллеров и permissions.
High Нет тестов. При 147 моделях и 180 endpoint-декораторах регрессии в auth, billing, EMR и scheduling сложно контролировать вручную.
Medium Frontend покрывает только core MVP. Большая часть backend-функциональности пока недоступна конечному пользователю через UI.
Medium TenantContextMiddleware не применен в `AppModule.configure`. Audit requestId и header tenant context могут не заполняться, если middleware не подключен на маршруты.
Medium RLS не реализован. Tenant isolation зависит от корректности всех Prisma filters в сервисах.
Medium Dashboard содержит статические KPI. Операционная панель показывает demo-метрики, не связанные с BI/reception API.
Medium Access token хранится в JS-readable cookie. Для production лучше пересмотреть cookie/session model и XSS exposure.
Low Некоторые документы описывают целевой стек шире текущего. В docs упомянуты GraphQL, ClickHouse, Kubernetes, Terraform, Prometheus, Loki, Sentry, но в коде этого пока нет.

12. Рекомендации для следующих ТЗ

Ниже темы, которые архитекторы могут использовать как входные точки для следующих задач.

  1. Привести `api-gateway` к полной карте backend API или явно разделить public/internal endpoints.
  2. Довести module registry: добавить `inventory-warehouse`, проверить зависимости и миграции seed.
  3. Сформировать тестовую стратегию: auth/session, scheduling conflicts, reception transitions, finance payments/refunds, EMR signing/amendment.
  4. Выделить UI roadmap для backend-модулей без интерфейса: org structure, EMR, finance, communications, analytics, inventory.
  5. Усилить multi-tenancy: подключить middleware, добавить RLS или централизованные Prisma guards/scopes.
  6. Закрыть production security gaps: rate limits, lockouts, password policy, MFA secret handling, cookie model.
  7. Синхронизировать docs/architecture с текущей реализацией и отметить target-only компоненты.
  8. Подключить реальные dashboard метрики из reception/analytics вместо статических frontend KPI.

Готовность для дальнейшего проектирования

Проект имеет достаточную доменную основу для продолжения архитектурного проектирования: данные и backend-контракты уже покрывают большинство ключевых контуров клиники. Главные следующие решения: public gateway/API boundary, UI coverage, тестирование и production hardening.
